# Backend Engineering Interview Questions

## 1. System Architecture

### Q1: What is the difference between Monolithic and Microservices architecture?
**Answer:**
- **Monolith:** All components (UI, database access, business logic) are bundled into a single unit and deployed together. It's easier to develop and test initially but becomes difficult to scale, deploy, and maintain as the codebase grows.
- **Microservices:** The application is broken down into small, independent services that communicate over APIs (usually HTTP or gRPC). Microservices allow independent scaling, fault isolation, and the ability to use different technologies per service. However, they introduce complexity in networking, data consistency, and distributed tracing.

## 2. API Design & Communication

### Q2: Compare REST, GraphQL, and gRPC. When would you use each?
**Answer:**
- **REST:** Uses standard HTTP methods (GET, POST, PUT, DELETE) and endpoints. Best for standard CRUD operations and public-facing APIs due to its widespread adoption and simplicity.
- **GraphQL:** A query language that allows clients to request exactly the data they need and nothing more. Best when serving diverse clients (e.g., mobile apps, web apps) that need varying data shapes, preventing over-fetching and under-fetching.
- **gRPC:** High-performance, low-latency framework using Protocol Buffers and HTTP/2. Best for fast, internal service-to-service communication in a microservices architecture.

## 3. Databases & Caching

### Q3: How do database indexes work, and what is the trade-off of having too many indexes?
**Answer:**
Indexes are database structures (usually B-Trees) that allow fast retrieval of rows without scanning the entire table (O(logN) instead of O(N)). 
**Trade-offs:** While they drastically speed up `SELECT` queries, they slow down `INSERT`, `UPDATE`, and `DELETE` operations because the index structure must be updated every time the underlying data changes. They also consume additional disk space.

### Q4: When and why would you use a caching layer like Redis? How do you handle cache invalidation?
**Answer:**
Redis is used as an in-memory data store to cache frequent, expensive, or slow database queries to reduce latency and database load.
**Cache Invalidation Strategies:**
- **TTL (Time to Live):** The cache expires automatically after a set time.
- **Write-around:** Application writes directly to the DB, and data is cached on the first read.
- **Write-through/Write-behind:** The application updates the cache and database simultaneously, or updates the cache and asynchronously updates the database.

## 4. Asynchronous Processing

### Q5: Why use Message Brokers (like RabbitMQ, Kafka, or BullMQ) instead of executing tasks synchronously via HTTP?
**Answer:**
Message brokers decouple the producer (sender) from the consumer (receiver), providing:
- **Resilience:** If the receiver is down, messages wait safely in the queue.
- **Scalability:** You can easily spin up more worker nodes to consume messages from the queue if the workload spikes.
- **Responsiveness:** The HTTP request can respond immediately (`202 Accepted`) while the heavy lifting (e.g., video processing, long AI generation) happens in the background.

## 5. Security

### Q6: How does JWT authentication work, and how do you handle token revocation?
**Answer:**
A JWT is a stateless, signed token generated by the server and sent to the client. The client sends it back in the `Authorization` header. Because it is stateless, the server does not need to query the database to verify it; it just verifies the cryptographic signature.
**Revocation:** Since JWTs are stateless, they cannot be natively revoked before expiration. To handle this:
1. Keep the JWT expiration short (e.g., 15 minutes).
2. Issue a longer-lived **Refresh Token** stored in the database.
3. If a user logs out, revoke the refresh token. The JWT will naturally expire soon, and the client will be unable to get a new one.
